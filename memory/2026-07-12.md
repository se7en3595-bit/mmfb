# 2026-07-12 任务进展与修复报告

今天完成了对分屏、拖拽调整大小、Excel字体颜色、预览复制、空数据表编辑以及 PDF 拖拽窗口消失等关键 Bug 的诊断与修复：

1. **分屏按钮点击修复**
   - 修复文件：`mmfb/frontend/js/bridge.js`
   - 根因：QWebChannel 并没有自动转换下划线命名为驼峰命名，JS 侧调用 `this._bridge.splitCurrentWindow` 为 undefined。
   - 解决方式：将调用改为正确的 `this._bridge.split_current_window`，并解析其返回 of JSON 数据。

2. **拖拽后窗口大小调整失效修复**
   - 修复文件：`mmfb/core/window.py`
   - 根因：通过 Win32 原生的 `DragAcceptFiles` 拖入文件时，接收 `WM_DROPFILES` 消息后，操作系统的鼠标捕获状态未正常释放，导致拖入文件后无法调整窗口大小。
   - 解决方式：在原生拖拽处理方法 `_on_native_drop_files` 结束前，主动调用 `ctypes.windll.user32.ReleaseCapture()` 释放鼠标捕获。

3. **Excel 字体颜色匹配修复**
   - 修复文件：`mmfb/handlers/xlsx_handler.py`
   - 根因：原本只提取了 cell 的 rgb 颜色。在 Excel 中如果使用了内置主题色（theme color）或者 indexed 颜色，其 rgb 返回为 None。此外，原本 load_workbook 设置的 `read_only=True` 也会使得 openpyxl 不加载 theme.xml。
   - 解决方式：重构 `_extract_style` 机制，解析 Indexed 颜色与主题色（包含 tint 亮度调节转换的精确公式），并将 read_only 参数改为 False。

4. **预览模式复制内容支持**
   - 修复文件：`mmfb/frontend/css/base.css`
   - 根因：`body` 统一设置了 `user-select: none;`，导致大多数预览器在普通预览模式下不能选择文本和复制。
   - 解决方式：专门针对正文容器 `.layout-main` 设置 `user-select: text` 样式，在预览状态下允许文本自由选中并复制。

5. **空数据表双击编辑功能**
   - 修复文件：`mmfb/frontend/js/xlsx_viewer.js`
   - 根因：原本的 `_renderEmpty()` 逻辑（无 Sheet 状态下）没有绑定双击编辑的事件监听。
   - 解决方式：当 xlsx 工作表个数为 0 或 rows/cols 为 0 时，双击空表提示背景即可初始化一个默认的 10x5 网格并渲染，并且对空单元格正确绑定了双击就地编辑与保存。

6. **拖入 PDF/Excel 边框消失与预览不显示修复（多轮迭代，最终彻底修复）**

   已确认的真实根因链：
   - **(根因 A) Router 双重重渲染**：`router.js` 的 `navigate()` 设置 `location.hash` 触发 `hashchange` → 自动调 `render()`，但代码里又手动在同函数内调用 `_parseHash() + render()`，导致每次路径变化时 handler 执行两次 → PDF/XLSX/Docx Viewer 双实例 → 第二个实例先清空 `root.innerHTML`（销毁第一个已渲染的 canvas/DOM），再重头加载 → 用户看到的视觉结果是"窗口边框消失 + 预览空白"。
   - **(根因 B) DWM 阴影易丢失**：Qt frameless 窗口通过 DWM 提供边框阴影（`DwmExtendFrameIntoClientArea({-1,-1,-1,-1})`），窗口状态变更（最小化/恢复、最大化/恢复）后 DWM 会自动重置帧扩展，阴影消失。
   - **(根因 C) 前几轮的"最小化阻断"修复走弯路**：曾尝试添加 `_drop_just_happened` 标志 + `QTimer.singleShot(1500)` 延迟放开 + nativeEvent 拦截 `WM_SYSCOMMAND/SC_MINIMIZE` + 在 `showMinpressed()` 中丢弃最小化调用，这些方案都会与新版本 QApplication event loop 产生不可预测的交互（窗口激活/焦度竞争）。

   最终修复：
   - `mmfb/frontend/js/router.js`: `navigate()` 改为检测 hash 是否真的变更：若未变才手动触发 render；若已变更则只设置 hash，让 hashchange 自动触发的 render 处理，杜绝 dual-render。
   - `mmfb/frontend/js/pages.js`: `view()` 入口加防重入守卫 `data-loading-file` attribute，同一 filePath 在首次未完成时二次调用直接 skip。
   - `mmfb/core/window.py`:
     * `showEvent()` 中调用 `_init_dwm_shadow()` 重注册 DWM 阴影，确保最小化恢复后边框仍可见。
     * 完全移除 `_drop_just_happened / _user_initiated_minimize` 状态变量。
     * 完全移除 `QTimer.singleShot` 延迟放开机制。
     * 完全移除 nativeEvent 对 `WM_SYSCOMMAND/SC_MINIMIZE` 的拦截。
     * 完全移除 `showMinpressed()` 中的阻断逻辑。
     * `_on_native_drop_files` / `dropEvent` / `load_file` 仅保留核心的 payload 发射逻辑。
   - 纯净后的信号处理：Python `filesDropped` 信号 → bridge.js `_bind_signals` → `MMFBNavigator._onFilesDropped` → `MMFBRouter.navigate` → 单次 `view()` → 单次 Viewer 初始化。

   请用户重新测试：分别拖入 PDF / Excel / Word，预期结果：
   - 窗口边框在所有状态下可见
   - 预览正常渲染
   - 可正常操作窗口

## 2026-07-12 分屏按钮卡死+边框丢失修复

**问题**: 点击分屏按钮后卡死约4秒，窗口边框丢失

**根因**:
1. 进入分屏时创建了2个全新的 QWebEngineView 实例，每个都要初始化 Chromium 渲染进程，导致 UI 冻结
2. 分屏切换时 `setCurrentWidget` 改变了窗口布局，DWM 阴影被重置但没有重新注册
3. 没有抑制标题栏自动隐藏，3秒定时器恰好在冻结期间触发

**修复文件**: `mmfb/core/window.py`
- `enter_split_mode()`: 入口调用 `_suppress_auto_hide_for(3000)` + 切换后调用 `_init_dwm_shadow()`
- `exit_split_mode()`: 入口调用 `_suppress_auto_hide_for(3000)` + 切换后调用 `_init_dwm_shadow()`
